# Motion UX コンテンツ戦略

## このドキュメントについて

「Motion UX とは何か」を **100本の記事と10本のデモ動画** で定義しきるための、12〜18ヶ月の制作計画。

前提：
- Pixels to Motors はブランド傘、Motion UX はその中の中核概念
- 既存環境（MacBook / ROS2 / Unity / mechArm 270-Pi / iPhone カメラ2台）で完結する範囲で組む
- DAY ログの現行ペースを活かして、追加負担を最小化する設計
- 日英バイリンガル運用が前提

---

## ② 実証系・DAYログ（60本）— 既に積み上がっている強み

DAY204+ の現状から、向こう1年で60本積める想定。**ただしログのままでは資産化しない**ので、以下の工夫を入れる。

**3つの仕掛け**
- 冒頭に必ず「これは Motion UX の〇〇という観点での実験」と一文入れる → ハブ記事への内部リンク
- 末尾に「今日得た判断軸」を1〜3行で抽出 → 将来の考察記事の素材になる
- 5〜10本ごとに「Log #1〜10 まとめ」記事を作る → ロングテールSEOの中継地点

**主要シリーズ**

| シリーズ | 内容 | 想定本数 |
|---|---|---|
| 動的追従シリーズ | Y軸 → X軸 → Z軸 → 統合 → 双腕 | 20本 |
| ドラッグティーチングシリーズ | 限界把握 → サンプル収集 → 模倣学習 | 15本 |

| その他 単発 DAY | 機材検証、SDK比較、コミュニティ動向など | 10本 |

## ③ 考察系（20本）— 思想を磨く記事

DAY ログから抽出した判断軸を、独立した記事に昇華するレイヤー。「設計思想を持つ実装者」というポジションを強化する。

**主要テーマ案（最初の10本）**

1. 制御の透明性 — Motion UX がブラックボックスを拒む理由
2. 精度より体験の成立 — 1mm と 10mm の間にあるもの
3. iPhone を入り口にする思想 — 機材コストと参入障壁
4. Motion UX における「止まり方」の設計
5. ロボットの動きと不気味の谷 — 速度プロファイルが感情に与える影響
6. Sim2Real ループを閉じる — 生成器・評価器・観測の3点セット
7. 双腕への分岐 — Motion UX の協調設計
8. シングルアームで分かること、双腕で初めて分かること
9. VLM を評価器にする時の限界と可能性
10. Motion UX におけるエラーリカバリの美学

**ペース：月2本 × 10ヶ月で20本完了**

## 環境特性を最大化する3つの工夫

### A. iPhone カメラ2台で「Motion UX 独自の撮り方」を定型化する

- 横カメラ：Y軸 / Z軸の軌跡を見る
- 上カメラ：X軸 / 接近距離を見る

この同時撮影は研究者がやらない構図。すべての動画で同じ視点配置を維持すると、見た人が「あ、Motion UX の動画だ」と一目で分かるようになる。**Pixels to Motors のビジュアル・アイデンティティ**として定型化する。

### B. Claude をハブ記事の編集相棒として常駐させる

定義系5本は月1回、エイイチさんの最新の理解で更新するのが理想。Claude に「前回からの変更点を反映してリライトして」と頼める状態にしておくと、メンテナンスコストが激減する。

### C. MacBook 上で記事 → 動画スクリプト変換のワークフローを作る

Claude Code で「DAYログ md → 動画スクリプト + ショート動画用テロップ案」を生成するスクリプトを用意しておくと、**1記事を3〜5プラットフォーム分のコンテンツに展開**できる。100本書く負担より、変換ワークフローを1つ作る方が、長期で見ると圧倒的に省力。

## 優先順位（最初の一手）

**ハブ記事「Motion UX とは何か」決定版を最優先で書く。**

この1本があると、残り99本すべての記事から内部リンクで戻る先ができて、SEO構造の中心が確定する。これがないまま DAY ログを増やしても、リンクが分散して権威が積み上がらない。

逆に言えば、これ1本を先に置いておけば、その後の99本はすべて「ハブに戻る経路」を持つ資産になる。
